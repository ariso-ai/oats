# In-App Updates — Design

**Status:** Approved (brainstorming)
**Date:** 2026-05-25
**Owner:** Ariso desktop

## Goal

Add a Sparkle-style in-app update experience to the Ariso Mac app: silent background checks, a polished "Update Available" dialog with release notes, and one-click install + relaunch. The app should behave like any other professionally-built Mac app (1Password, Bear, Things) when a new version ships.

## Decisions

| Area | Decision |
|---|---|
| Distribution source | GitHub Releases (existing) — `latest/download/latest.json` |
| Channels | Stable only (beta channel is a future addition, not in scope) |
| Check cadence | On launch (after 10s) + every 24h while running |
| User controls | Install Update / Remind Me Later / Skip This Version |
| Mandatory updates | Supported via a `mandatory: true` flag in the manifest |
| Download behavior | Prompt first, then download on user confirmation (classic Sparkle) |
| Dialog style | Modern card: centered hero with app icon, version chip, release notes in a soft card |
| Implementation | Tauri's official `@tauri-apps/plugin-updater` + `@tauri-apps/plugin-process` |

Each decision and the alternatives considered are below.

## 1. Architecture

```
┌─ Mac client ──────────────────────────────────────┐
│                                                   │
│  ┌─ Rust (Tauri) ──────────────────────────────┐  │
│  │  tauri-plugin-updater  (Ed25519 verify,     │  │
│  │                         download, install)  │  │
│  │  tauri-plugin-process  (relaunch)           │  │
│  │  update_manager.rs                          │  │
│  │    • startup check (10s delay)              │  │
│  │    • hourly tick, runs check at most /24h   │  │
│  │    • commands: update_check,                │  │
│  │      update_install_and_relaunch,           │  │
│  │      update_skip_version, update_snooze,    │  │
│  │      update_get_state                       │  │
│  │    • events: update://checking,             │  │
│  │      update://available, update://none,     │  │
│  │      update://error,                        │  │
│  │      update://download-progress             │  │
│  │    • state in settings.json:                │  │
│  │      auto_check_enabled,                    │  │
│  │      skipped_version, snoozed_until,        │  │
│  │      last_check                             │  │
│  └─────────────────────────────────────────────┘  │
│                                                   │
│  ┌─ Vue frontend ──────────────────────────────┐  │
│  │  UpdateView.vue  (route /#/update)          │  │
│  │     — opened on-demand by update_manager    │  │
│  │  SettingsView → About section               │  │
│  │     — version, status, manual check,        │  │
│  │       auto-check toggle                     │  │
│  │  Tray menu → "Check for Updates…"           │  │
│  └─────────────────────────────────────────────┘  │
│                                                   │
└──────────────┬────────────────────────────────────┘
               │ HTTPS
               ▼
┌─ github.com/ariso-ai/conflux/releases ────────────┐
│  latest/download/latest.json                      │
│  latest/download/Ariso.app.tar.gz                 │
│  latest/download/Ariso.app.tar.gz.sig             │
│  latest/download/Ariso_*.dmg   (first-install)    │
└───────────────────────────────────────────────────┘
```

Three components:

- **`update_manager.rs`** owns scheduling, state, and command-handling. Single source of truth.
- **Update window** is a Vue route created on-demand by `update_manager` when an update is found. Closes itself after action; not pre-created.
- **Settings → About** and **tray menu** are passive surfaces: they send commands and listen for events.

The frontend never schedules, never persists state. All policy lives in Rust.

## 2. Release artifact + signing

The signed DMG continues to be produced for first-install. Three new things are added.

### 2.1 Ed25519 keypair (one-time setup)

```bash
npx tauri signer generate -w ~/.tauri/ariso-updater.key
```

- **Private key** → added to the `release` GitHub Environment as secret `TAURI_SIGNING_PRIVATE_KEY` (and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` if a password is set).
- **Public key** → committed in `tauri.conf.json` under `plugins.updater.pubkey`.

This is separate from Apple's Developer ID certificate. Apple's signature proves *who built the bundle*; the Ed25519 signature proves *Tauri's updater can trust the bytes before swapping the .app*. Both are required end-to-end.

### 2.2 Release assets

After `tauri:build` with the signing env vars set, the bundler emits two additional files alongside the DMG:

- `Ariso.app.tar.gz` — the updater installs by extracting this over `/Applications/Ariso.app`
- `Ariso.app.tar.gz.sig` — the detached Ed25519 signature

### 2.3 `latest.json` manifest

Generated by the release workflow after the build step and uploaded as a release asset:

```json
{
  "version": "0.3.0",
  "notes": "<contents of the GitHub release body>",
  "pub_date": "2026-05-25T18:00:00Z",
  "mandatory": false,
  "platforms": {
    "darwin-aarch64": {
      "signature": "<contents of Ariso.app.tar.gz.sig>",
      "url": "https://github.com/ariso-ai/conflux/releases/download/v0.3.0/Ariso.app.tar.gz"
    }
  }
}
```

`mandatory` is a custom field. The Tauri updater ignores it; the frontend reads it from the raw manifest body to decide whether Skip/Later actions render. Marking a release mandatory after the fact is done by editing the manifest in the release's assets (or by including `[mandatory]` in the release title, which the workflow parses).

### 2.4 Config changes

**`tauri.conf.json`** adds:

```json
"plugins": {
  "updater": {
    "endpoints": ["https://github.com/ariso-ai/conflux/releases/latest/download/latest.json"],
    "pubkey": "<base64 Ed25519 pubkey>"
  }
}
```

Why pin to `/latest/download/` rather than a versioned URL: GitHub serves the asset from whichever release is currently marked "Latest". Rolling back is then a one-click operation in the GitHub UI (mark an older release as latest) with no client logic.

**`Cargo.toml`** adds:

```toml
tauri-plugin-updater = "2"
tauri-plugin-process = "2"
```

**`package.json`** adds:

```json
"@tauri-apps/plugin-updater": "^2",
"@tauri-apps/plugin-process": "^2"
```

### 2.5 CI workflow changes (`.github/workflows/desktop.yaml`)

The existing `release` job is extended:

1. Add `TAURI_SIGNING_PRIVATE_KEY` and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` to the job's `env:` from `release` environment secrets.
2. After the `Build, sign, and notarize` step, add a new step that generates `latest.json`:
   - reads the GitHub release body (via `github.event.release.body`) for `notes`
   - reads the `.sig` file contents
   - constructs the JSON manifest
   - writes it to `latest.json`
3. Extend the existing `softprops/action-gh-release@v2` step's `files:` glob to upload `Ariso.app.tar.gz`, `Ariso.app.tar.gz.sig`, and `latest.json` in addition to the DMG.

The `validate` job is unchanged — it doesn't have signing secrets and doesn't build a bundle.

## 3. `update_manager.rs` — orchestration

A single Rust module that owns all update state. Spawned as a Tokio task from `main.rs` setup.

### 3.1 State

```rust
struct UpdateState {
    last_check: Option<DateTime<Utc>>,
    latest_known: Option<UpdateInfo>,
    auto_check_enabled: bool,             // persisted; default true
    skipped_version: Option<String>,      // persisted; cleared on newer version
    snoozed_until: Option<DateTime<Utc>>, // persisted; set by "Remind Me Later"
    current_check: Option<JoinHandle<()>>, // dedupes concurrent checks
}

struct UpdateInfo {
    version: String,
    notes: String,
    mandatory: bool,
}
```

Persistence uses the existing `tauri-plugin-store` `settings.json` (already used for `recordingMode`). Keys: `update.auto_check_enabled`, `update.skipped_version`, `update.snoozed_until`, `update.last_check`.

### 3.2 Commands

| Command | Behavior |
|---|---|
| `update_check(force: bool)` | Runs a check. `force=true` ignores `auto_check_enabled`, `snoozed_until`, **and `skipped_version`** — the manual menu item always re-prompts. |
| `update_install_and_relaunch()` | Downloads the update bytes, installs, calls `process::restart()`. |
| `update_skip_version(version: String)` | Persists `skipped_version`. Rejects if `latest_known.mandatory`. |
| `update_snooze()` | Sets `snoozed_until = now + 24h`. Rejects if `latest_known.mandatory`. |
| `update_get_state() → UpdateState` | Snapshot of current state for the Settings view. |

### 3.3 Events (emitted to all windows)

| Event | Payload | Consumer |
|---|---|---|
| `update://checking` | `{}` | Settings: show spinner |
| `update://available` | `{ version, notes, mandatory }` | `update_manager` opens UpdateView; Settings updates status |
| `update://none` | `{}` | Settings: "You're up to date" |
| `update://error` | `{ message }` | Settings: inline error |
| `update://download-progress` | `{ downloaded, total }` | UpdateView: drives progress bar |

### 3.4 Scheduling

```rust
// pseudo
spawn(async move {
    sleep(Duration::from_secs(10)).await; // don't fight startup
    loop {
        if should_check(&state).await {
            check(&state).await;
        }
        sleep(Duration::from_secs(3600)).await;
    }
});

fn should_check(s: &UpdateState) -> bool {
    s.auto_check_enabled
        && s.last_check.map_or(true, |t| Utc::now() - t > Duration::hours(24))
        && s.snoozed_until.map_or(true, |t| Utc::now() > t)
}
```

The hourly wake-up keeps the loop simple; sleeping for 24h directly doesn't survive sleep/wake reliably, while a short tick + a cheap predicate does.

### 3.5 Skip semantics

Two ways the skip is bypassed:
- When an automatic check returns a version *greater* than `skipped_version`, the skip is cleared and the user is prompted. So skipping `0.3.0` does not suppress `0.3.1`.
- A manual check (`force=true`) ignores `skipped_version` entirely — the user explicitly asked, that overrides their earlier dismissal.

An automatic check that returns a version *equal* to `skipped_version` results in `update://none` to the Settings view (no dialog re-opens).

### 3.6 Mandatory updates

When `latest_known.mandatory == true`:
- `update_skip_version` and `update_snooze` reject with an error.
- The frontend doesn't render those buttons (defense in depth, but the backend enforces).
- The UpdateView blocks close via `CloseRequested` interception.

The user can still quit Ariso entirely from the menu bar; on next launch the prompt returns. Mandatory does not block the app from running while offline — it only blocks dismissing the dialog when the dialog is shown.

### 3.7 Concurrency

`current_check` (a `JoinHandle`) ensures only one check runs at a time. A second `update_check` call while one is in flight awaits the existing one rather than starting a new one.

### 3.8 Auto-check failure handling

Network errors, 404s, and parse errors in the *automatic* path are logged to stderr and suppressed from the user. A private retry-backoff cap ensures the loop won't hammer the server when offline (at most one attempt per hour). Errors from the *manual* path always surface in Settings.

## 4. Frontend surfaces

### 4.1 Update window (`UpdateView.vue`, route `/#/update`)

Created on-demand by `update_manager` when `update://available` fires:

```rust
WebviewWindowBuilder::new(app, "update", WebviewUrl::App("/#/update".into()))
    .title("")
    .inner_size(420.0, 360.0)
    .resizable(false)
    .center()
    .always_on_top(false)
    .build()?;
```

If the window already exists, `get_webview_window("update")` is used to focus it instead of creating a duplicate.

**Layout (modern card):**
- 64×64 app icon (rounded corners, soft shadow) centered at top
- "Update Available" title
- Version chip (`0.3.0`) + "You have 0.2.0" subtitle
- Release notes rendered as Markdown in a soft gray card (`#f5f5f7` background, ~78px min height, scrolls if longer)
- Action row at bottom: Skip and Later as quieter text links, "Install Update" as the primary button

**Action behavior:**

- **Install Update** → `invoke('update_install_and_relaunch')`. Button morphs into a determinate progress bar driven by `update://download-progress`. Skip/Later are hidden during download. On completion the OS replaces the app and `process::restart()` is called; the window dies with the process.
- **Remind Me Later** → `invoke('update_snooze')` → `getCurrentWindow().close()`.
- **Skip** → `invoke('update_skip_version', { version })` → `getCurrentWindow().close()`.

**Mandatory variant:**
- Skip and Later links are not rendered.
- Window-close intercepted via `on_window_event` (`api.prevent_close()`).

**Download error mid-install:** button resets to "Install Update", inline message under the action row: *"Couldn't download update. Try again?"*. No silent failure on the manual path.

### 4.2 Settings → About section (`SettingsView.vue`)

Replaces the existing `Ariso v{{ appVersion }}` text:

```
About
┌────────────────────────────────────────────┐
│  Ariso 0.2.0                               │
│  You're up to date.  Last checked: 2m ago  │
│                                            │
│                       [ Check for Updates ]│
│                                            │
│  ☑ Automatically check for updates          │
└────────────────────────────────────────────┘
```

Three states for the status line:
- *"You're up to date. Last checked: X ago"* — green check icon
- *"Update available: 0.3.0"* + **[ Show Details ]** button (opens or focuses the update window) — accent color
- *"Checking…"* with subtle spinner — while a check is in flight

The checkbox writes `auto_check_enabled` via `tauri-plugin-store` `settings.json`. Unchecking does not cancel an in-flight check but suppresses the next scheduled one.

Mounted hook calls `update_get_state()` to populate; listens for `update://*` events to stay live while open.

### 4.3 Tray menu (`tray.rs`)

Add a single `Check for Updates…` menu item. The click handler lives in Rust (same as the other tray items) and calls `update_manager::check(force = true)` directly — no round-trip through the frontend.

No badge, no tray-icon decoration, no notification — when a check finds something, the dialog window appearing centered + frontmost is the notification. (A tray-icon dot is a possible future addition but not in scope.)

## 5. Testing

The plugin handles the cryptographic + extraction paths; we test our orchestration.

### 5.1 Manual end-to-end (the only test that proves it works)

Set up a throwaway `conflux-update-test` GitHub repo. Publish `v0.2.1-test` there with `latest.json`, `Ariso.app.tar.gz`, and `.sig`. Install v0.2.0 locally, point `tauri.conf.json` at the test repo via a `test-updater` Cargo feature, and verify:

1. Dialog appears within 10s of launch.
2. **Install Update** → progress bar → relaunch as v0.2.1.
3. **Skip This Version** → next manual check immediately offers it again (skipped version is recorded but `force=true` overrides).
4. **Remind Me Later** → no auto-check fires for 24h.
5. Mandatory flag (`mandatory: true` in manifest) → no Skip/Later buttons render; close button disabled.
6. Bad signature (corrupt the `.sig` file) → install fails with the verification error in Settings.

### 5.2 Unit tests on `update_manager`

Pure functions on `UpdateState` — fast and meaningful:

- `should_check` truth table (auto-check off, recently checked, snoozed, all combinations).
- Skip semantics: skipping `0.3.0` then receiving `0.3.1` re-prompts.
- Snooze expiry: `snoozed_until` in past → next check fires.
- Settings persistence round-trip via the store plugin.

### 5.3 No tests on download/install

That's the plugin's job. Mocking it would prove nothing.

### 5.4 CI impact

`npm run vite:build` continues to pass with the new view. No new CI machinery for updater testing — it's manual by nature.

## 6. Error handling

| Failure | Where | Behavior |
|---|---|---|
| Network down (auto-check) | `update_manager` loop | Log, retry next interval, never surface |
| Network down (manual check) | Settings | Inline: *"Couldn't reach update server. Check your connection."* |
| Bad/missing signature | Plugin verify step | Log + Settings error: *"Update verification failed. Please reinstall from ariso.ai."* — refuse to install (real attack signal) |
| Download interrupted | Mid-install | Reset button to *Install Update*, inline message: *"Download interrupted. Try again?"* |
| `latest.json` malformed | Parse step | Log + generic Settings error. Treat as "no update" for the auto-path. |
| Installed version > manifest version | `update_manager` | Treat as "up to date" — handles dev-installed newer builds |
| App quit mid-download | OS | Tauri's downloader writes to a temp file; partial file abandoned. Next check restarts cleanly. |

## 7. Edge cases

- **First run after update.** Freshly-installed version writes `last_check = now` so it doesn't immediately re-check on launch. Avoids the "update every launch for a while" feel if releases land back-to-back.
- **Manual downgrade (dev builds).** Semver comparison via the plugin's `Version`; if local > remote, no prompt.
- **Sleep / wake.** Tokio's timer uses CFRunLoop on Mac and wakes on resume — no special handling.
- **Mandatory + offline.** App still loads and works. Mandatory only blocks dismissing the dialog *when shown*; it does not block usage when no manifest is known.
- **Two windows know about the same update.** UpdateView and Settings both listen for `update://available`. UpdateView is a singleton (re-creation is replaced by focus). Settings just updates its status line.
- **Hidden `main` window.** It also receives events. Harmless — no UI bound to them.

---

## 8. Out of scope

- **Beta channel.** Stable only. Future addition: a second manifest URL + a Settings toggle.
- **Per-architecture builds.** `darwin-aarch64` only; the existing self-hosted runner is Apple Silicon and matches Ariso's user base.
- **Delta updates.** Full tarball replacement.
- **Rollout percentages, kill switch, per-user gating.** Would require migrating from `latest/download/latest.json` to a server-rendered manifest on `api.ari.ariso.ai`. Pinning to a single URL in `tauri.conf.json` makes this migration a one-line change later.
- **Notification toast / tray badge.** Dialog window is the notification.
- **Windows / Linux builds.** App is Mac-only.

---

## 9. Approaches considered

1. **Tauri's official updater plugin (chosen).** ~2-3 days of work. All cryptography, extraction, and install handled. Single owner (Tauri team) for the security-sensitive code.
2. **Sparkle.framework via Swift FFI.** Native Mac dialog, but requires writing a Tauri plugin wrapper around `SPUStandardUpdaterController`, bundling `Sparkle.framework` into the `.app`, and re-doing Tauri's signing infra. ~1-2 weeks; the gain (a native dialog) is negative since the chosen design replaces it anyway.
3. **Roll our own.** Fetch manifest, verify signature, swap bundle, relaunch. Reinvents Tauri's updater for no gain and adds a security-critical maintenance surface.
